Что такое статическая память и динамическая память в контексте программы?
p.s. существует статическая оперативная и динамическая память, оперативная и динамическая типизация и тому подобное. Здесь конкретно будет информация про программную память, то есть та память, которая выделяется во время запуска программы и во время ее выполнения.

Статическая память выделяется во время компиляции программы. Прямо в момент компиляции, например, уже известен размер массива и переменной, и где они будут находиться в памяти.

Динамическая память подразумевает, что эта та память, которая будет выделяться в момент выполнения программы, то есть на старте программы мы ничего не можем сказать об этом сегменте, например, где он будет выделен.

Динамическая память - эта та память, которая может расширяться. А статическая память расширяться не может, она фиксированного размера и в течение программы менять свой размер не может.

#### Виртуальное адресное пространство программ в конкретных ОС
Программа состоит из нескольких сегментов: сегмент кода (текста), сегмент данных, BSS, куча, стек. Но помимо пяти этих основных сегментов, которые формирует сам язык C для выполнения программы, ОС привносит в это виртуальное адресное пространство программы свои сегменты, которые необходимы для запуска исполняемого файла (программы) в контексте конкретной ОС.

Как выглядит программа при запуске в LInux в 32 разрядном режиме.
![[Pasted image 20260218162925.png]]
4-ёх гб сегмент виртуального адресного пространства делится на 3 нижних гб user space (наши программы) и верхний 4-ый гб kernel space. Так у каждого процесса в Linux, сколько бы мы процессов не запустили, нижнюю часть (частичку) будет занимать виртуальное адресное пространство программы, а верхний гб будет отображать виртуальное пространство процесса в само ядро. Без этого никак не обратиться с системным вызовам. 

Весь код, что находится в пространстве ядра, является высокопривилегированным (нулевое кольцо защиты - Ring 0), а в User space - непривилегированный уровень (Ring 3). Чтобы перейти из user space в kernel space, у нас есть механизм поддержки машинных инструкций для реализации системного вызова, чтобы выполнить какой-то системный вызов и вернуться обратно.

То есть помимо того, что у нас в памяти теперь в виртуальном адресном пространстве находится наша программа со своими секциями, в наше же адресное простарнство, причем в фиксированные адреса, отображается само ядро.

![[Pasted image 20260218164946.png]]
Пример: есть 3 процесса - Firefox, команда ls, Firefox. У всех программ разное виртуальное пространство. Как видно, пользовательское пространство у всех разнится, и на физическое пространство будет отображается в разные места.

![[Pasted image 20260218165430.png]]
ОС также еще привносит сегмент между стеком и кучей - отображение, которое используется для динамических библиотек. Динамическая память будет находится в куче или в области memory mapping segment, там же, где библиотеки. Если выделяется сегмент памяти больше 128 кб, то он выделяется не в куче, а в memory mapping segment. Но ссылки будут находиться в стеке: указатель, который будет указывать на выделенную динамическую область, будет находиться в стеке, а сами данные в куче или в memory mapping segment.

#### Как работать с динамической памятью?
```c
#include <unistd.h>

int brk(void* addr); // работает по абсолютному адресу

void *sbrk(intptr_t increment); // работает инкрементно -> +-50 -> +-50 байт

// Увеличивают границу кучи (подымают вверх) либо уменьшают (вниз, сокращают ее до нуля).
// Когда увеличиваем, мы выделяем память, а ОС делает там, что в нашем виртуальном пространстве появляются новые страницы, для которых нужно создать отображение виртуального адреса на физический - это ОС делает автоматически, когда она будет реализовывать эти две функции.
```
Эти 2 функции являются deprecated, они настолько запрещены, что их нельзя вызвать даже в качестве теста.
Их цель - управление верхушкой кучи (сегмент, который динамично растет, как и стек, в большую и меньшую стороны). Как только программа начнет разрастать свою кучу и стек до такой степени, что они встретятся -> мы исчерпали все виртуальное адресное пространство и, соответственно, всё физическое пространство. Любые попытки программы попросить больше памяти не сработают и на этом все закончится.

Все системные и библиотечные вызовы нужно проверять на ошибки. Как? Использовать мануалы. В мануалах все описано, что возвращает функция, если та завершилась успешно или некорректно. А затем просто, уже при программировании, после вызова функций пишем условие, которое будет делать проверку на возрат ошибок.

```c
#include <unistd.h>

int main(void)
{
	char* ptr1;
	char* ptr2;
	
	ptr1 = sbrk(100);
	ptr2 = sbrk(20);
	...
	sbrk(-120);
	return 0;
}
```

Чем пользоваться, если `brk() и sbrk()` - это одна сплошная заморочка? Тем более если их даже нельзя использовать?
Ответ: `malloc(), calloc(), realloc()`
Важная ремарка: реализаций malloc в glibc несколько, `#include <malloc.h> - более свежая и адекватная`, а также `#include <stdlib.h>`. 

`#include <malloc.h>` - более потокобезопасная
```c
#include <malloc.h>
#include <stdlib.h>
#include <alloca.h>

// выделяет КОЛИЧЕСТВО БАЙТ, которые передаются в аргумент функции.
void* malloc(size_t size); // malloc(sizeof(int) * 5); -> выделилось 20 байт. Возвращает адрес на начало области.
// malloc, по сути, просто двигает границу либо дает на переиспользование высвобожденный кусок -> в памяти всегда будет мусор.

// вариация malloc с инициализацией, зануляет область.
void* calloc(size_t nmemb, size_t size); // указываем кол-во элементов и размер одного элемента -> calloc(5, 4); -> 20 байт.
// возвращает адрес на начало выделенной области в куче.

// если выделенная область больше не нужна, просто высвобождаем эту область.
void free(void* ptr); // free(array);
// если два раза высвободить одну и ту же область, это приведет к ошибке. Обязательно нужно высвобождать, чтобы не нагружать ОС + хороший тон.

// изменяет размер, например, массива. Размер может быть как и меньше прошлого, так и больше.
void* realloc(void* ptr, size_t size); // realloc(array, 5);

// выделение динамической памяти в стеке (в момент выполнения программы)
void* alloca(size_t size); // не стоит использовать, выделять память под большой массив в стеке - плохая идея.

// ТЕПЕРЬ ФУНКЦИИ ПО СТАТИСТИКЕ (для получения информации во время выполнения программы)

void malloc_stats(void);

struct mallinfo mallinfo(void);

void malloc_trim(size_t pad);

int mallopt(int param, int value);
```

Когда мы используем библиотеку `malloc.h`, она хранит данные в куче в виде чанков. Когда мы выделяем память, она выглядит как на картинке ниже. Помимо, допустим, выделенных 100 байт (user data), будут еще 3 поля cur_size, prev_size. Все это вместе называется чанком. Функция `malloc()` вернет на начало user data, чтобы мы могли вписать данные. Но под этим указателем и в конце области user data находится служебная информация, которая используется функцией `free()`. Почему два поля cur_size? Когда блок выше выделится, верхнее cur_size на самом деле будет полем prev_size для этого следующего блока. Это нужно для того, чтобы когда мы высвобождали чанк, мы могли бы посмотреть не свободен ли чанк ниже с целью объединения их в один большой чанк и увеличить шанс того, что мы сможем этот чанк потом переиспользовать. Для чего это? Чтобы не двигать кучу, ведь так мы будет у системы память забирать, лучше же переиспользовать чанки ниже.

#### Утечка памяти
```c
#include <malloc.h>
#include <unistd.h>
#include <alloca.h>

int main(void)
{
	char* ptr;
	
	ptr = malloc(10); // после ptr = malloc(20); нельзя будет высвободить, адрес потерян
	...
	ptr = malloc(20); // 10 байт не устраивает, нужно 20
	...
	free(ptr); // высвободится ptr = malloc(20);
	free(ptr); // приведет к segmentation fault
	
	return 0;
}
```
