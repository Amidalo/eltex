В Linux есть два типа библиотек: статические и динамические. Речь пойдет про статические.

Линовка программы (компиляции объектных модулей и библиотек) происходит на самой последней 4 стадии - объединение объектных модулей в исполняемую программу. Здесь включается редактор связей и его задача скомпоновать программу в один единственный бинарник.

Статическая библиотека - это набор объектных модулей в Linux, которые представлены в виде одного архива - файл, в котором объектные модули идут друг за другом и вначале есть индекс, который говорит в каком смещении от начала файла какой объектный модуль находится. По имени объектного модуля мы понимаем какая функция внутри него реализована: у каждой функции свой .c файл. Зачем? Когда мы объединяем статическую библиотеку в архив, и когда мы будем компоновать библиотеку с нашей программой, нужно будет взять из архива только те объектные модули, функции которых мы используем. Библиотеки оформаляются статически таким образом, чтобы мы могли из нее изъять только ту часть, которую мы, действительно, используем в программе, а остальное не трогать.

#### Что такое объектный модуль?
Объектный модуль мы получаем на 3 стадии компиляции, когда происходит трансляции из ассемблера в машинный код, а на выходе мы получаем набор машинных инструкций, который можно запустить, если самим расположить их в памяти и настроить у процессора регистры, чтобы он начал выполнять нужную нам программу, которая была загружена в память нами же. Но этого недостаточно, чтобы выполнить этот бинарник в контексте ОС, для этого и нужна 4 стадия - мы на объектный модуль надеваем заголовок исполняемого файла, чтобы загрузчик исполняемых файлов знал что нужно с этим бинарником сделать, чтобы его можно было выполнить в контексте ОС.

Но что детально представляет из себя объектный модуль?
Любой объектный файл состоит из следующих 4 секций -> symbol table, relocation table, text (содержит машинный код), data (содержит глобальные инициализированные переменные).
![[Pasted image 20260227003205.png]]
Вспомним, что в модели памяти у нас есть еще BSS (глобальные неинициализированные переменные), куча и стек - это все то, что относится к языку C. Эти секции появляются во время загрузки бинарника, но информация о том, какого размера должна быть куча, где она должна быть расположена, какого размера должен быть стек и где он должен быть расположен, расположена в заголовке исполняемого файла.

Благодаря таблице символов мы можем знать где в начале секции TEXT какая функция начинается и какие глобальные переменные начинаются в секции DATA.

Таблица релокации нужна линковщику, чтобы потом пробежаться в результирующей секции, которая будет скомпонована из всех секций text объектных модулей, которые будут объединены в один. Другими словами, в этой таблице релокации хранятся смещения в секции text машинных инструкций, которые нужно потом в момент линковки подправить, иначе ничего работать не будет.

#### Как происходит объединение двух объектных модулей?
![[Pasted image 20260227004719.png]]
Исходный код файла file2.c:
```c
#include <stdio.h>
#include "file1.h"

int main(void)
{
	long a, b, c, d, e, f, g, h, i, j;
	
	a = 5;
	b = 6;
	c = 7;
	d = 8;
	e = 9;
	f = 10;
	g = 11;
	h = 12;
	i = 13;
	
	printf("main=%p test=%p\n", main, test);
	j = test(a, b, c, d, e, f, g, h, i);
	printf("sum = %ld\n", j);
	
	while(1)
	{
		j++;
	}
	
	return 0;
}
```
Исходный код файла file1.c:
```c
long test(long a, long b, long c, long d, long e, long f, long g, long h, long i)
{
	long sum = 0;
	
	sum = a + b + c + d + e + f + g + h + i;
	
	return sum;
}
```
В file1.o будет храниться только секция text1, а секции data1 не будет, потому что наша функция ничего не выводит на экран, мы только считаем, никаких глобальных переменных не используем. В функции никакие другие функции не вызываются, поэтому таблица релокации будет пустая (править ничего не нужно). Таблица символов будет состоять только из одного символа, который будет говорить, что в секции text1 функция file1.c будет начинаться с нулевого смещения (с самого начала секции text1). Вот весь объектный модуль, который получится в результате доведения file1.c до 3 стадии.

В file2.c вызывается функция main(), в text2 будет реализация только функции main, в text2 будет находиться машинная инструкция, которая будет вызывать функцию из file1.c, но мы не знаем где она сейчас находится -> резервируем место в text2, в relocation table2 заносим об этом месте смещения информацию о том, что когда мы узнаем адрес функции из file1.c, нам надо будет вернуться в text2 и подправить адрес, так как там сейчас нулевой адрес. Symbol table2 будет состоять из двух записей - запись самой функции main, которая будет говорить, что main начинается с нулевого смещения секции text2 и неопределенный символ file1.c, о котором мы пока ничего не знаем, смещения никакого не будет.

Теперь два этих модуля мы передаем линкеру, который объединит их в одну исходную программу. Секции text и data объектных модулей объединятся в один - будет text3 и data3 (если бы были инициализированны глобальные переменные). Таблица символов теперь может скорректироваться, потому что теперь у нас есть file1 и file2 в одной секции text3, теперь они находятся по совсем другим смещениям, не с нулевого, а после секции text1 (речь о file2), а вот file1 с нулевого смещения.
![[Pasted image 20260227011110.png]]
Теперь имея таблицу релокации второго объектного модуля, пойдем в секцию text2 и скорректируем адрес, так как мы в объединенной секции text3 знаем где у нас находится file1, то есть смещение -> получили полноценный бинарник.

#### Как происходит объединение программы со статической библиотекой?
Статическая библиотека - это, по сути, архив, а архив есть файл, в котором все файлы идут друг за другом в линией, и, по сути они представляют этот единый файл.
![[Pasted image 20260227012054.png]]
Исторически сложилось, что в таком виде удобно хранить архивы, раньше, пока не появились жесткие диски и всякие флеш-носители, админы записывали все на магнитную ленту, и был индекс, который говорил по какому смещению в этом файле какой файл находится, чтобы потом из этого архива можно было извлечь эти файлы по отдельности или сразу их все получить на выходе как отдельные файлы. Не надо путать архив с сжатым файлом, архив не сжимается, он просто представляется в виде одного файла.

Статические библиотеки - архивы, имеют расширение .a - от слова archive и префикс lib- - по нему сразу ясно, что это библиотека. Индекс хранит где у libstdio.a расположена функция printf(), которая реализуется отдельным объектным модулем, и так далее. Как мы выяснили ранее, каждый объектный модуль содержит секцию text (если, конечно, библиотека не содержит только глобальную переменную, то этой секции не будет), возможно содержит секцию data, relocation table и symbol table. Когда мы компонуем объектный модуль нашей программы, file1.o, где вызывалась функция, например, print() из библиотеки, то результирующий бинарник заберет из библиотеки только объектный модуль той функции, которую мы использовали. Конечно, может быть такое, что функция может вызывать какую-то другую библиотечую функцию, тогда и объектный модуль той функции тоже будет частью бинарника.

#### Плюсы и минусы статических библиотек
Статика удобна в том случае, когда мы хотим распространяться одним бинарником без каких-либо задержек, потому что динамические библиотеки подгружаются в момент выполнения программы -> runtime система языка должна потратить дополнительные накладные расходы на загрузку библиотек в память. В случае статических библиотек этого делать не нужно, статических код максимально быстрый.

Теперь представим, что у нас есть два бинарника, оба вызывают функцию printf(). Вот мы загрузили в память два бинарника -> в памяти будут находиться две копии функции printf(). При использовании статических библиотек мы будем много дублировать, если у нас есть куча бинарников, которые используют одни и те же функции статической библиотеки, то когда мы все эти бинарники загрузим в память, у нас будет неимоверно большое количество дублежа кода.

#### Как собирается статическая библиотека
```c
gcc file1.c -c // доводим до объектных модулей
gcc file2.c -c

// Можно узнать информацию о таблице символов (symbol table)
nm file1.o

// Можно узнать информацию о таблице релокации (relocation table)
readelf -r file1.o

// Можно узнать информацию о секциях text и data
objdump -D file1.o

ar rc libmyname.a file1.o file2.o // с помощью утилиты ar создаем архив в виде библиотеки, которую можно затем распространять людям, которые будут использовать мой код, и рядом с библиотекой всегда будет заоловочный файл, который содержит прототипы функций, иначе программа, которая использует ваши функции, не сможет скомпилироваться до объектного модуля, так как мы знаем, чтобы в синтаксис языка C отработал, надо как минимум знать как правильно вызывать наши функции, а чтобы ее знать, нам нужен интерфейс (прототип), а это может быть только в заголовочном файле -> БИБЛИОТЕКА ВСЕГДА РАСПРОСТРАНЯЕТСЯ С ЗАГОЛОВОЧНЫМИ ФАЙЛАМИ.

// c - create (создать библиотеку), r - replace (заменить - удобно использовать, если в файле .c мы что-то поменяли и затем перекомпилировали объектный модуль. С помощью этого параметра мы подменяем конкретный модуль, а если те уже содержатся в библиотеке, они просто встанут на место старого объектного модуля)

gcc test.c -o test -L. -lmyname
// Для компоновки статических библиотек обычно используется 2 ключа:
// -L. с указанием папки где находится библиотека. Если не указать (и укажем только имя библиотеки), то мы не скажем где искать, то компилятор пойдет искать в стандартных путях. Компилятор скажет, что такой библиотеки нет и компиляции не пройдет. Точка после -L означает текущий каталог
// -l - указываем имя библиотеки без префикса lib- и расширения .a

./test
```
