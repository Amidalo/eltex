#### О системном программировании
Системеное ПО включает в себя оболочку, текстовый редактор, комплиятор, отладчик, утилиты ядра и системные демоны. Эти компонеты работают с кодом ядра и бибилотеками C.

---
Cистемное программирование держится на:
1] системных вызовах;
2] библиотеке C;
3] компиляторе C.

---
**Системные вызовы** - обращение к функциям из пользовательского пространства (текстовый редатор, игра и т.д.) в ядро с целью запросить у ОС службу или ресурс. В Linux реализовано меньше системных вызовов, чем в ядрах других ОС. В ядре Linux для каждой из архитектур (Alpha, i386, PowerPC) реализован свой собственный список системных вызовов. Тем не менее очень большой поднабор системных вызовов реализован одинаково во всех архитектурах.

Приложения из user space не могут напрямую выполнять код ядра или манипулировать данными ядра из-за соображений безопасности. Вместо этого приложению предоставляется механизм, по которому оно может сигнализировать ядру о том, что нужно выполнить системный вызов. Приложение отправляет прерывание ядру и исполняет только тот код, к которому ему разрешило обратиться ядро.

Реализация механизма зависит от архитектуры. В архитектуре i386 приложение user space выполняет инструкцию программного прерывания `int0x80` - делает переход в пространство ядра - защищенную область ядра, где ядро исполняет обработчик программного прерывания. А обработчик для прерывания `0x80` это обработчик системного вызова. Приложение сообщает ядру, какой системный вызов нужно выполнить и с какими параметрами, используя машинные регистры.

Системные регистры обозначаются номерами начиная с 0. Чтобы запросить системный вызов в i386, приложение из user space помещает номер системного вызова в регистр `eax` и только после этого исполняет инструкцию `int0x80`. А передача параметров обрабатывается схожим образом: в i386 для каждого из возможных параметров используется отдельный регистр (`ebx, ecx, edx, esi, edi`) - содержат в указанном порядке первые 5 параметров. Бывает, что системный вызов включает в себя более 5 параметров, тогда используется один регистр, указывающий на буфер в user space, в этом буфере хранятся все параметры вызова. У большинства системных вызовов не более двух параметров.

Как системному программисту, обычно не нужно знать как ядро обрабатывает выполнение системных вызовов. Все это закодировано в стандартных соглашениях о вызовах для каждой архитектуры и автоматически обрабатывается компилятором и библиотекой C.

---
В современных системах Linux **библиотека C** - это GNU libc (glibc). Находится в самом сердце приложений Unix. Если программировать на другом языке, то скорее всего библиотека C тоже принимает участие, обернутая высокоуровневыми библиотеками. Помимо реализации стандартной библиотеки C, glibc предоставляет обертки для системных вызовов, поддержку многопоточной обработки и базовые возможности приложений.

---
В Linux стандартный **компилятор C** - это коллекция компиляторов GNU (GNU Compiler Collection, gcc). Первоначально gcc был cc (C Compiler), разрабатывался в проекте GNU. gcc может также расшифровываться как GNU C Compiler, если говорить в контексте языка C. Но все же, так как была добавлена поддержка многих других языков, под gcc обычно имеется ввиду коллекция компиляторов GNU. 

gcc - это также и двоичный файл, он вызывает компилятор C. В книге Лава 'Системное программирование на Linux' под gcc обычно имеется ввиду программа, если контекст не подразумевает другого.

Сам компилятор C тесно связан с системным программированием, так как он помогает реализовывать стандарт C и системный интерфейс ABI.

#### Интерфейсы API/ABI
API (интерфейс прикладного уровня) и ABI (двочиный интерфейс приложений) - наборы определений и описаний, влияющие на переносимость программ.

---
**API** обычно предоставляет набор функций, которые какая-то часть ПО (обычно высокоуровневая, но не всегда) может вызывать из другой части ПО (обычно низкоуровневой). Пример: есть концепция вывода текста на экран, а API может абстрагировать эту концепцию при помощи семейства функций, эти функции обеспечивают все необходимое для этой концепции. В API просто определяется интерфейс, а часть ПО, которая предоставляет интерфейс API, называется реализацией API (implementation).

API называют контрактом, но это неверно, ведь API - это не двухстороннее соглашение. Тот, кто использует API (обычно высокоуровневое ПО) не может ничего внести в этот API и его реализацию.
