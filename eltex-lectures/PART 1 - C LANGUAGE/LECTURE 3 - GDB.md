Отладчик нужен для того, чтобы выполнить программу по шагам и обнаружить какую-то ошибку. Мы компилируем программу и в отладчике анализируем уже исполняемый код. 

В отладчике мы можем выполнять программу как и пошагово, так и непрерывно, устанавалить точки останова на конкретное место в программе с целью быстрого обнаружения ошибки.

Обычными `printf()` отлаживать программу не очень удобно, так как не до конца понятно в каком указателе произошла ошибка. Отладчик сразу показывает ту строку, которая виновата в этой ошибке. В отладчке можно посмотреть значение самого адреса и что расположено по этому адресу. В отличие от `printf()` мы можем сходить прямо в ячейку памяти в отладчике (так как в нем мы получаем доступ ко всему адресному пространству процесса) и убедиться в том, действительно ли то лежит в этой ячейке или нет.

#### Почему именно GDB?
1] В gdb можно отлаживать удаленно: можно запустить программу отладки на компьютере, а саму программу, которую мы отлаживаем, на другой железке с другой архитектурой. Главное, чтобы было сетевое соединение. 
На железке может и не быть графического интерфейса, самого отладчика и других инструментов для сборки программы -> на самой железке работать неудобно. Поэтому мы собираем программу у себя на компе под эту железку, скопировал на нее, запустил и начал отлаживать. В этом случае всегда можно будет, в случае ошибки, переписать программу, перекомпилировать ее и отправить обратно на железку.

2] gdb способен сохранять трассы: во время анализа программы можно сохранить все свои действия в виде log-файла, например, делать обратную перемотку, то есть выполнять программу пошагово назад (отлаживание в обратном порядке).

3] gdb поддерживать backtrace: это, например, когда мы можем увидеть стек всех вызовов функций, и нам интересно как мы попали в какую-то функцию, начиная с функции `main()` и какие параметры были переданы в эту функцию.

4] С помощью отладчика мы можем поменять значения переменной в живой программе. Также мы можем менять машинные инструкции.

#### Попрактикуемся! (без отладочной информации)
Есть программа:
```c
#include <stdio.h>

void my_func(void)
{
	int c = 0;
	c = c + 5;
}

int main(void)
{
	int* ptr = NULL;
	
	printf("Hello!\n");
	my_func();
	*ptr = 5;
	
	return 0;
}
```
Если скомпилировать ее `gcc main.c -o main` и запустить `./main`, то выведится `Hello!`, а после возникнет `Segmentation fault (core dumped)`. Дело в том, что в начале мы указали, что указатель ни на что не указывает, то есть = NULL, а затем, никак его не инициализировав, начали разыменовывать.

Теперь запустим gdb `gdb ./main`.
Что будет, если отлаживать программу без отладочной информации? Отладчик не знает, что программа написана на языке C, он видит только машинный код. Когда мы загрузили программу, она находится в преостановленном состоянии, то есть она не выполняется. Чтобы она начала выполняться, нужно выполнить команду запуска.

Для запуска используется команда `run` или `r`). Если запустить прямо сейчас, то программа будет выполняться непрерывно, так как ни одна точка останова не была установлена.

Обычно, при использовании отладчика, мы видим строку исходного кода, на которой мы остановили выполнение программы. Мы не видим ее, потому что мы скомпилировали программу без отладки. Такая скомпилированная программа может отлаживаться в отладчике, но она отлаживается в машинных инструкциях.

Отличие команд `next`, `step`, `continue`.
Когда мы достигнем строку, где вызывается функция, то:
1] `next` не будет входить в функции, а выполнит ее так, будто мы сделали это за 1 шаг.
2] `step` позволит войти в функцию и отлаживать ее пошагово.
Если надоело выполнять программу пошагово, то используйте команду `c` или `continue`.

#### Попрактикуемся! (с отладочной информацией)
Есть программа:
```c
#include <stdio.h>

void my_func(void)
{
	int c = 0;
	c = c + 5;
}

int main(void)
{
	int* ptr = NULL;
	
	printf("Hello!\n");
	my_func();
	*ptr = 5;
	
	return 0;
}
```
Выполним команду `gcc main.c -o main -g`. Флаг `-g` позволяет добавить отладочную информацию.

---
ВАЖНЫЙ НЮАНС: 
Если мы собираем программу таким образом:
`gcc main.c -o main -g` (из файла .c сразу в бинарник), тогда флаг `-g` стоит в конце. 

Если мы сначала создаем объектный модуль: 
`gcc main.c -c` ->   main.o
А потом бинарник:
`gcc main.o -o main -g` <- ничего не выйдет, отладочной информации нет (No debugging symbols found in main).

ПРАВИЛЬНО будет указать флаг `-g` во время создания объектного модуля:
`gcc main.c -c -g` ->   main.o
А потом бинарник:
`gcc main.o -o main`
При `gdb main` -> отладочная информация появится (вы не увидите No debugging symbols found in main).

Чем отличается программа с отладкой?

Без отладочной информации:
`b main` -> `Breakpoint 1 at <address>`
Высвечивается только адрес и никакой информации о том, где в файле находилось данная функция.

С отладочной инфомацией:
`b main` -> `Breakpoint 1 at <address>: file main.c, line <N>`
Теперь показывается какой файл и строка.

Теперь если сделать `run`, мы будем видеть исходный код программы. Если выполнить команду `list`, то можно увидеть полный листинг функции `main()`. Мы также можем работать с переменными, используя команду `p` -> `p ptr`, получим $1 = (int *) 0x0.

Если встречается функция (например, `printf()`) какой-либо библиотеки (например, glibc), то нужно понимать, что это машинный код, который прилинковался без отладки -> нежелательно делать `step` (войти в функцию), так как мы будем работать с машинным кодом. При работе с машинным кодом использовать команду `next` не надо, так как она работает с теми строчками, который написал человек. Нужно использовать команду `next instruction` -> `ni` (выполнение по инструкции). Если функция большая, то можно просто прописать `fini` -> завершить текущую функцию. Этим мы вернемся из функции `printf()` в программу.

Если встречается функция, которая является частью нашего бинарного файла (и там содержится отладочная информация), мы можем войти в функцию командой `step` для проверки.

Изменение переменных.

Для этого использовать команду: `set variable`
`set variable c=5`

Через команду `set` можно менять не только переменные, но и адреса. Можно узнать адреса машинных инструкций и через `set` по указанию адреса изменить значение машинной инструкции.

Можно также вывести значению по адресу. Допустим у нас тип int c = 17;
`x/4db <address>` -> 17 0 0 0
`x/4xb <address>` -> 0x11 0x00 0x00 0x00
x -> 
4 - поставили после /, сколько байт надо вынести; 4, так как int = 4 байта.
d - формат, десятиричный, восьмеричный и т.д. d = десятиричный.
b - как вывести: побайтово, по словам или по двойным словам и т.д. b = побайтово.
address - с какого адреса начиная.
