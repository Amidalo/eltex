#### Подробнее о кадре функции
Как известно, первые 6 параметров функции передаются через регистры, а остальные попадают в стек в область аргументов функции. Есть нюанс: если анализировать машинный код, то хоть первые 6 параметров передаются через регистры, но они копируются в область сразу за областью локальных переменных -> получается, они все равно попадают в кадр: первые 6 хранятся в конце, а остальные в области аргументов функции.

Область локальных переменных тоже не всегда выделяется, так как если функция является листовой (другую функцию не вызывает), то обычно память в стеке не выделяется, так как сам стек в фиксированном размере доступен (2 мб? по разному на разных ОС). 

#### Практика
Исходный код файла main.c:
```c
#include <stdio.h>
#include "func.h"

int main(void)
{
	long a, b, c, d, e, f, g, h, i, j;
	
	a = 5;
	b = 6;
	c = 7;
	d = 8;
	e = 9;
	f = 10;
	g = 11;
	h = 12;
	i = 13;
	
	printf("main=%p test=%p\n", main, test);
	j = test(a, b, c, d, e, f, g, h, i);
	printf("sum = %ld\n", j);
	
	while(1)
	{
		j++;
	}
	
	return 0;
}
```

Исходный код файла func.c:
```c
long test(long a, long b, long c, long d, long e, long f, long g, long h, long i)
{
	long sum = 0;
	
	sum = a + b + c + d + e + f + g + h + i;
	
	return sum;
}
```

gcc -fno-stack-protector -no-pie main.c -c -g
gcc -fno-stack-protector -no-pie func.c -c -g
gcc main.o func.o -o test
./test main=0x401136 test=0x40120f
./test main=0x401136 test=0x40120f
-> адреса не меняются, но если убрать -no-pie, то будут.

Между кучей и стеком находится область, в которую загружаются динамические библиотеки, пока что используется одна динамическая библиотека - glibc. 

Инструкция call автоматически подставит в кадр функции адрес возврата, но саму эту подстановку мы не увидим, так как это происходит автоматически. Он сохраняет текущее значение регистра rip в этот адрес возврата. Но после функции мы должны будем возвращаться уже на следующую инструкцию.
